NoSQL Design Document — Smart E-Commerce System
==================================================

1. Purpose
----------
This document describes a NoSQL data model for storing product reviews
and activity logs, comparing it with the relational approach used in
the PostgreSQL implementation.

2. NoSQL Data Model (Document-Oriented — MongoDB)
--------------------------------------------------

JSON Document Structure for Reviews:

{
  "review_id": "REV-10023-uuid",
  "product_id": 101,
  "user_id": 5,
  "rating": 4,
  "title": "Amazing Laptop Pro",
  "comment": "The performance is outstanding. Handles video editing with ease.",
  "images": [
    "https://cdn.ecommerce.com/reviews/rev-10023-img1.jpg",
    "https://cdn.ecommerce.com/reviews/rev-10023-img2.jpg"
  ],
  "helpful_votes": 42,
  "tags": ["performance", "display", "high-end"],
  "verified_purchase": true,
  "created_at": "2026-04-08T10:00:00Z",
  "updated_at": "2026-04-08T12:30:00Z"
}

3. Justification for NoSQL (Reviews & Logs)
---------------------------------------------

NoSQL document databases (like MongoDB) are better suited for reviews and
activity logs for the following reasons:

(a) FLEXIBLE SCHEMA: Reviews often have variable fields — some users attach
    images, others don't; some include tags, helpful votes, or titles. In a
    relational model, these optional fields would either require many nullable
    columns or separate junction tables. A document model naturally handles
    this variability by storing each review as a self-contained JSON document
    with only the fields that are present.

(b) HIGH WRITE VOLUME: Activity logs are written continuously as users browse,
    search, and interact with the system. NoSQL databases are optimized for
    high write throughput with eventual consistency, making them ideal for
    logging scenarios where immediate strong consistency is not required.

(c) DENORMALIZATION FOR READ PERFORMANCE: In the relational model, displaying
    a review with product and user information requires JOIN operations across
    multiple tables. In a document model, denormalized data (such as embedding
    the product name within the review document) eliminates JOINs entirely,
    providing faster reads for customer-facing pages where reviews are displayed.

(d) HORIZONTAL SCALABILITY: As the e-commerce platform grows, the volume of
    reviews and logs will increase rapidly. NoSQL databases support horizontal
    scaling (sharding) more naturally than relational databases, allowing the
    system to distribute data across multiple servers without complex
    partitioning schemes.

4. Hybrid Approach (PostgreSQL + JSONB)
-----------------------------------------

PostgreSQL supports JSONB columns, which offer a middle ground:
- Relational integrity (foreign keys to Products and Users tables)
- Flexible JSON storage for variable review fields
- GIN indexes on JSONB columns for efficient querying

This approach could be used when the team wants to maintain a single
database technology while still accommodating semi-structured data.

5. Comparison Summary
---------------------

| Aspect              | Relational (PostgreSQL)     | NoSQL (MongoDB)             |
|---------------------|-----------------------------|-----------------------------|
| Schema              | Fixed, normalized           | Flexible, denormalized      |
| Reviews storage     | Requires nullable columns   | Natural document fit        |
| Read performance    | JOINs required              | Single document read        |
| Write performance   | ACID transactions           | High throughput, eventual   |
| Scalability         | Vertical (scale up)         | Horizontal (scale out)      |
| Data integrity      | Strong (FK, CHECK, UNIQUE)  | Application-level           |
| Query language      | SQL                         | MongoDB Query Language      |
